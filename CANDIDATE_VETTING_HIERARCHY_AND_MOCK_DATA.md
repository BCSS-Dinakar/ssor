# SSOR — Automated Candidate Vetting Hierarchy & Mock Database Guide

> **Detailed Technical & Operational Documentation** for the automated 3-Level Priority Hierarchy Scan, the 20 CCTNS + 20 ePetty Cases Mock Database, Risk Tier Color Badges, and Live UI Vetting Integration.

---

## 📌 Executive Summary

To strengthen background verification checks for organizations (schools, creches, care homes, transport operators) while preventing unauthorized disclosures, the SSOR platform now implements a **real-time, PostgreSQL-backed automated criminal scan engine (`POST /api/police/verifications/:id/scan`)**.

When an organization submits a vetting request for a prospective employee (`CandidateVerification`), and the Police Officer clicks **Process Vetting $\rightarrow$ Start Criminal Register Check** inside their restricted console, the system queries live criminal registries (`CCTNS` and `ePetty Cases`) in strict accordance with a **2-Stage Precedence & 3-Level Match Hierarchy Engine**.

---

## 🏗️ 1. Database & Schema Precedence (`CCTNS` First $\rightarrow$ `ePetty Case` Second)

The verification engine strictly adheres to data source priority:

```
[Candidate Verification Request Submitted]
                   │
                   ▼
       ┌───────────────────────┐
       │ Step 1: Scan CCTNS    │  ◄── Primary National/State Criminal Database
       │ (20 Mock Records)     │      (FIRs, Convictions, Serious Sexual Offences)
       └───────────┬───────────┘
                   │
         Match Found? (Matches > 0)
          /                 \
        YES                  NO
        /                     \
       ▼                       ▼
[Return CCTNS Matches]   ┌───────────────────────┐
                         │ Step 2: Scan ePetty   │  ◄── Secondary Petty Offence Database
                         │ (20 Mock Records)     │      (Eve-teasing, Nuisance, Harassment)
                         └───────────┬───────────┘
                                     │
                           Match Found? (Matches > 0)
                            /                 \
                          YES                  NO
                          /                     \
                         ▼                       ▼
                [Return ePetty Matches]    [Return Clean Vouched Status]
```

### Why Precedence Matters:
1. **CCTNS (`CctnsRecord`)** stores high-severity criminal offences (FIRs, Sessions Court convictions, POCSO Act violations). If a candidate matches a CCTNS record, that record takes immediate priority over any minor infractions.
2. **ePetty Cases (`EPettyCase`)** stores minor infractions, public nuisance, stalking warnings, and fine-paid disposals. The engine scans ePetty cases **only if** zero matches are found across CCTNS.

## ⚡ 2. The 3-Level Match Hierarchy

Within the active database layer being scanned, the engine executes three progressive passes to classify match precision (`Priority Hierarchy`):

| Hierarchy Level | Matching Rule | Description & Operational Handling |
| :--- | :--- | :--- |
| **High Priority** | **Exact Name AND Phone Match**<br>`(name === candidateName && phone === candidatePhone)` | **Highest Confidence Match.** The candidate exact full name and verified mobile number match a criminal or petty offence record. Immediate manual inspection and rejection advised. |
| **Medium Priority** | **Exact Name Match Only**<br>`(name === candidateName && phone !== candidatePhone)` | **Moderate Confidence Match.** The candidate name matches an existing offender record, but the phone number differs (could indicate a new SIM/number change). Officer must cross-verify father's name, age, and alias. |
| **Low Priority** | **Exact Phone Match Only**<br>`(phone === candidatePhone && name !== candidateName)` | **Investigation Required.** The phone number submitted by the candidate belongs to an existing registered offender under a different name (indicates potential identity concealment or recycled phone number). |

## 🎨 3. Risk Tier Color Badges (`Risk Tier System`)

Every suspect card generated by the live database scan dynamically renders visual **Risk Tier Badges** to instantly alert the vetting officer to the severity of the offence:

* 🔴 **Red Tier (`Tier: Red`)**: High-risk sexual assault, POCSO Act convictions, or repeat grave offences.
* 🟠 **Orange Tier (`Tier: Orange`)**: Stalking, workplace harassment, or unsolicited sexual advances (`EPC-2025-088: Imran Shaikh`).
* 🔵 **Blue Tier (`Tier: Blue`)**: Public voyeurism, privacy breach, or trespassing into protected premises.
* ⚫ **Black Tier (`Tier: Black`)**: Aggravated cyber stalking, criminal intimidation, and digital exploitation.
* 🌸 **Pink Tier (`Tier: Pink`)**: Habitual eve-teasing, minor harassment, and transport misconduct.
* 🟢 **Green Tier (`Tier: Green`)**: First-time minor public nuisance or resolved/fine-paid infractions.

## 💻 5. Frontend UI Integration (`VerificationVetting.js`)

### Key Improvements Made:
1. **Removal of Hardcoded Overrides (`hasHistory` logic removed):**
   * Previously, the frontend checked whether `candidateName` contained hardcoded surnames (`Reddy`, `Bai`) before showing match results.
   * Now, `startCriminalCheck` directly queries `policeApi.scanVerificationById(id)`.
   * If `suspects.length > 0`, the UI **guarantees** displaying the matched suspect cards.
   * If `suspects.length === 0`, the UI displays the green **Vetting Clearance Vouched** block (`Candidate has no police records...`).

2. **Dossier Inspection & Decision Workflow:**
   * Officers can click **Inspect Record Dossier (`Eye Icon`)** on any matched card to open the complete criminal history sidebar (showing FIR Number, Police Station, Court Name, and Disposal/Conviction Date).
   * Once cross-verified, officers click **Select Match (`Confirm Suspect`)** and proceed to make the official clearance determination (**Reject Vetting Clearance** or **Approve Clearance**).
